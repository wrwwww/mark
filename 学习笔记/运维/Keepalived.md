# Keepalived是什么

**Keepalived** 是一款开源的 Linux 路由软件，主要用来为 Web 服务、负载均衡器等关键业务提供**高可用性**和**负载均衡**解决方案。

它的核心思想，是通过虚拟IP（VIP）的“漂移”来解决服务器的单点故障问题，保障系统持续稳定运行。

### 核心功能与工作原理

简单来说，Keepalived 主要靠以下两大“法宝”协同工作：

- **VRRP 协议实现高可用（故障转移）**：这是 Keepalived 的看家本领。
    1.  **组建集群**：将多台服务器（通常是主、备两台）组成一个虚拟路由器组，共享一个对外的虚拟 IP（VIP）。
    2.  **心跳检测**：**Master（主服务器）** 会定期通过 VRRP 协议向 **Backup（备用服务器）** 发送“心跳”广播包，告知自己一切正常。
    3.  **自动切换**：如果 Backup 在一段时间内收不到心跳包，就会判断 Master 发生故障，并立即接管 VIP 和服务，成为新的 Master。整个过程自动完成，对于客户端来说完全无感知。

- **健康检查机制**：为了保证服务质量，Keepalived 会从三个层面定期检测后端服务器的健康状况：
    - **Layer3 (网络层)**：通过发送 ICMP 请求（就是 `ping` 命令），检查服务器的 IP 地址是否存活。
    - **Layer4 (传输层)**：通过尝试连接服务器的特定端口（如 Web 服务的 80 端口），判断服务是否可用。
    - **Layer5-7 (应用层)**：这是最精细的检查。它会向应用发一个 HTTP 请求（如 `GET /`），然后校验返回的内容（如通过 MD5 校验），确保应用不仅能连通，并且功能正常。

### 核心架构

Keepalived 在运行时，会启动三个独立的进程来保证自身的稳定与高效：
- **父进程（Watchdog）**：一个监控者。它像一个看门狗，专门负责监控下面两个子进程的健康状况。
- **VRRP 子进程**：专门负责 VRRP 协议的通信，处理 Master 的选举和 VIP 的漂移。
- **健康检查子进程**：专门负责执行各类健康检查任务，监控后端服务器的状态。

这种拆分设计，避免了健康检查的偶然卡死影响到核心的 VRRP 故障切换功能，提高了系统的鲁棒性。

### 典型应用场景

Keepalived 通常不单独使用，而是与负载均衡器或 Web 服务器结合，构建高可用集群。

| 场景 | 技术组合 | 解决的问题 |
| :--- | :--- | :--- |
| **高可用负载均衡集群** | **LVS + Keepalived** | 这是最经典的组合。LVS 负责分发流量，Keepalived 为 LVS 调度器提供高可用。当主调度器宕机，备机立刻接管，保证入口不中断。 |
| **高性能 Web 集群** | **Nginx/HAProxy + Keepalived** | 让多台 Nginx 服务器共享一个 VIP。主 Nginx 故障时，备 Nginx 连同 VIP 一起接管，守护 Web 服务入口的稳定。 |

### 基础配置解析

Keepalived 所有的功能都通过 `/etc/keepalived/keepalived.conf` 配置文件实现。以下是一个极简的配置示例，可以帮你快速理解：

```bash
# 在 Master 服务器上的配置
vrrp_instance VI_1 {          # 定义 VRRP 实例
    state MASTER              # 初始状态为 Master
    interface eth0            # VIP 绑定的网卡
    virtual_router_id 51      # 虚拟路由 ID，主备必须一致
    priority 101              # 优先级，数值越高越优先
    advert_int 1              # 心跳包发送间隔（秒）
    virtual_ipaddress {       
        192.168.1.100/24      # 虚拟 IP (VIP)，供客户端访问
    }
}
```

配置中的 `priority`（优先级）是选举 Master 的关键，数值越大越容易被选为主。备份服务器（Backup）只需将 `state` 改为 `BACKUP`，并将 `priority` 设置得更低即可。
